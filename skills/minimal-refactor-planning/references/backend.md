# 后端改造参考

当问题主要集中在 handler、route、service、job、worker、脚本、CLI 命令或多步骤服务端流程时，读取这个 reference。

## 常见后端异味

### 1. handler / controller 管太多
典型信号：

- 请求解析
- 参数校验
- 权限判断
- 流程编排
- 外部 API 调用
- 持久化
- 响应组装

全都混在一个函数里。

优先方向：

- 协议/请求相关的东西留在边缘
- 业务主流程下沉到 service / use-case
- 外部集成放到明确边界后面

### 2. 同一套业务流程出现在多个入口
典型信号：

- 多个 route handler、cron job、consumer、command 里在复制相似逻辑
- 不同副本之间已经开始轻微漂移

优先方向：

- 把业务主流程集中起来
- 各入口只负责适配自己的输入输出

### 3. 基础设施泄漏到处都是
典型信号：

- 数据库、队列、HTTP client、日志和业务判断全混在一起
- 不启动整套环境就几乎无法验证核心决策逻辑

优先方向：

- 把流程编排和基础设施适配拆开
- 收出一条可用 stub/fake 验证的核心路径

### 4. util 泛滥导致隐藏耦合
典型信号：

- 到处都是不知归属的 utils
- 行为依赖副作用或共享可变状态
- 改一处，别处也容易被波及

优先方向：

- 按职责分组，而不是按“通用工具”命名
- 用边界清晰的模块代替松散 util 堆

### 5. 容易冲动做大重构
典型信号：

- 因为代码乱，就想顺手把 route、service、repository、domain model 全改一遍

优先方向：

- 只围绕本次要改的路径重构
- 先解决当前最影响理解和复用的问题
- 一旦这次需求变得顺畅，就及时收手

## 如何选抽象

### service / use-case
适合承接业务主流程和流程编排。

### adapter / gateway
适合隔离外部系统：

- database
- third-party API
- queue
- storage
- RPC client

### helper / module
适合抽出确定性强、接近纯函数的逻辑。

### command wrapper / job runner
适合让多个运维/任务入口共用一套执行流程。

## 后端问题回答时应说清楚什么

一个好的后端改造方案，通常要明确：

1. 哪些内容留在协议边缘
2. 哪些内容下沉到 service / use-case
3. 哪些外部依赖需要隔离
4. 哪些行为保持不变
5. 如何在不扩大范围的前提下验证

## 常用表述

- “把请求/协议相关逻辑留在边缘，业务流程往里收。”
- “这些入口应该变成同一条执行路径上的薄适配层。”
- “这样可以隔离外部 I/O，而不用顺手引爆整个架构。”
- “对这次需求来说，抽一个聚焦的 service 就够了。”

## 常见坑

默认不要这样建议：

- 为局部清理直接做一整套 DDD 重构
- 没有即时收益却硬上 repository/service/factory 多层
- 只有一个调用方就先做过度抽象
- 重构后仍把协议层和业务语义混在一起
- 一边改行为，一边大规模重命名，导致风险叠加
