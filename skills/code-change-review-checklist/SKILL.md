# Code Change Review Checklist

这个 skill 用来把“写完就算了”的开发习惯，变成**实现前先判断落点，实现后再做结构复盘**。

核心目标不是挑语法毛病，而是提前发现这些更容易积累技术债的问题：

- 代码写错文件
- 职责边界混乱
- 新逻辑塞进旧文件导致文件继续膨胀
- 边界情况没覆盖
- 业务值、路径、状态、文案硬编码
- 架构虽然能跑，但已经开始发臭

## 适用场景

当用户出现这些诉求时使用本 skill：

- 想把“需求开发自检流程”沉淀成规范
- 想做一个通用代码 review / 自查 skill
- 想让 AI 在改代码前先判断文件职责和落点
- 想在开发完成后系统检查边界情况、硬编码、坏味道
- 想降低“先糊上去再说”式实现
- 想把工程经验提炼成可复用 checklist

## 使用时机

这个 skill 应该在两个时间点使用：

### 1. 动手前
先判断“这段代码该写哪”。

### 2. 改完后
再检查“虽然实现了，但有没有留下债”。

不要只在收尾时做 review。很多结构问题，应该在落代码之前就拦住。

## 核心工作流

收到一个实现需求后，按下面顺序检查。

### 第一步：判断文件职责
先回答这些问题：

1. 当前准备修改的文件是干什么的？
2. 这个需求的代码，是否真的应该放在这个文件里？
3. 这次改动是在扩展原职责，还是在偷偷混入一个新职责？
4. 如果新建文件，会不会比继续塞进当前文件更清晰？
5. 如果新建文件，应该放在哪一层、哪个目录、以什么命名出现？

判断标准：

- **同一抽象层优先放一起**：视图逻辑、状态逻辑、数据访问、工具函数，不要乱混。
- **同类职责优先靠近**：如果项目里已有同类模块，优先跟已有模式对齐。
- **增长趋势要考虑**：如果这次只是“再加几十行”，但明显还会继续长，应该尽早拆分。
- **调用方向要自然**：不要为了图省事，让底层依赖上层。
- **避免工具文件垃圾场化**：不是所有不好放的代码都该进 `utils`。

常见判断信号：

应该继续放在当前文件：
- 只是补足已有职责
- 改动和文件现有语义高度一致
- 不会让文件认知负担明显变大

应该拆新文件：
- 引入了独立逻辑块，可以单独命名
- 会被复用，或未来高概率继续扩展
- 让主文件变得更短、更清晰
- 可以形成稳定边界，如 `hooks`、`services`、`types`、`helpers`、`components`

应该换个位置放：
- 逻辑层级不对
- 依赖方向反了
- 当前文件只是“能放进去”，不是“应该放进去”

如需更细的落点判断模式，读：`references/file-placement-patterns.md`

### 第二步：检查边界情况
代码能跑，不等于需求完整。

至少检查这些边界：

- 空值 / null / undefined
- 空数组 / 空对象 / 缺字段
- 加载中 / 失败 / 超时 / 重试
- 用户重复点击 / 并发触发 / 防抖节流
- 权限不足 / 状态不允许 / 前置条件不满足
- 极小值 / 极大值 / 临界值
- 国际化、时区、格式差异
- 数据源异常、接口字段变化、后端返回不规范
- 组件卸载、异步回调晚到、状态竞争

不要只问“正常流程通不通”，要问：

- 不合法输入会怎样？
- 没数据时会怎样？
- 请求失败会怎样？
- 用户乱点会怎样？
- 上下游返回脏数据会怎样？

如需更系统的边界检查维度，读：`references/edge-case-catalog.md`

### 第三步：检查硬编码
硬编码不只是写死字符串。

重点看这些内容是否被不合理写死：

- 状态值、枚举值
- 路由路径、跳转地址
- 接口地址、环境域名
- 权限码、角色名
- 时间范围、分页大小、重试次数
- magic number
- 业务规则阈值
- 文案、提示语、标题
- 样式尺寸、颜色值
- fallback 逻辑中的默认值

判断原则：

可以接受的硬编码：
- 领域内稳定常量
- 明确的协议字段
- 不会变且具有共识的标准值

应该抽离的硬编码：
- 未来可能变化
- 被多处复用
- 带业务语义但看不出来源
- 会因环境、角色、配置而变化
- 改一次需要全局搜字符串

抽离方向通常包括：
- 常量文件
- 配置项
- 枚举 / 映射表
- i18n 文案资源
- 服务端配置或后端下发
- 主题变量 / design token

### 第四步：检查架构坏味道
不是所有“能跑”的实现都值得留下。

重点识别这些坏味道：

- 一个文件同时承担页面、状态、请求、转换、校验多个职责
- 上层业务细节泄漏到底层通用模块
- 为了复用而过度抽象，反而更难懂
- 同一逻辑复制三份，只改了变量名
- 数据流绕来绕去，很难追踪来源
- 组件过大，局部修改需要理解整页上下文
- 参数越来越多，函数签名已经说明边界不合理
- if/else 分支爆炸，说明缺少更稳定的数据结构或策略模式
- 临时补丁叠在旧补丁上，没人敢动底层
- 命名越来越模糊，如 `handleData`、`tempList`、`flag2`

遇到坏味道时，不一定要立刻大重构，但至少要明确：

1. 这是不是当前需求顺手就该修的？
2. 如果现在不修，会不会显著放大后续成本？
3. 能不能做一个低风险、局部的结构修正？
4. 如果暂时不修，是否应明确记录为后续债务？

## 输出要求

使用本 skill 时，输出尽量按下面结构组织。

### 简版输出
适合中小需求：

1. **建议改动落点**：改当前文件 / 拆新文件 / 挪到别处
2. **理由**：一句话说清职责判断
3. **边界提醒**：本次最容易漏掉的 2~5 个边界
4. **硬编码提醒**：本次最需要抽离的硬编码项
5. **架构提醒**：是否已有坏味道，要不要顺手收拾

### 详细版输出
适合复杂需求或 code review：

- 文件职责判断
- 推荐放置方案
- 可选拆分方案
- 边界情况清单
- 硬编码清单
- 架构坏味道清单
- 是否建议本次顺手优化
- 如果优化，建议最小改法是什么

## 决策原则

### 原则 1：优先最小正确结构，不要最小可运行结构
“先跑起来再说”只适合一次性脚本，不适合长期维护代码。

### 原则 2：抽象要晚于重复，但不要晚于失控
不要一开始就抽象，也不要等复制三四次还不收。

### 原则 3：边界处理不是补丁，是实现的一部分
漏边界，不叫“后续优化”，很多时候那就是未完成。

### 原则 4：硬编码的风险看变化概率和扩散范围
不是所有硬编码都错，但高变化、高扩散的一定要收敛。

### 原则 5：坏味道不一定本次全修，但必须被看见
最糟糕的不是有债，而是没人意识到债已经形成。

## 和其他 skill 的关系

- 如果任务是“把模糊需求转成完整开发方案”，优先结合 `ai-spec-coding`
- 如果任务重点是“先给最小改法，再决定要不要动代码”，可和偏规划类 skill 结合
- 本 skill 更偏**实现前后的结构审视与质量把关**，不是需求拆解本身

## 默认行为

当本 skill 被触发时：

1. 先判断代码落点是否合理
2. 再补边界情况检查
3. 再检查硬编码
4. 最后评估是否已有架构坏味道
5. 输出时优先给明确结论，不绕

如果信息不足，也优先基于现有上下文给出“当前最合理判断”，而不是把问题原样丢回给用户。
