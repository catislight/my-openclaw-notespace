# AI Spec Coding

这个 skill 用来把“AI 写代码”从随缘对话，变成可复用的工程协作流程。

核心思想只有一句话：

**先消除不确定性，再让 AI 高速执行。**

不要默认所有需求都要写完整 spec。先判断任务颗粒度，再选协作模式。

## 适用场景

当用户出现这些诉求时使用本 skill：

- 想做一个通用的 AI 编程协作流程
- 想把需求、PRD、设计稿、接口文档转成开发方案
- 想建立团队级规范，让 AI 输出更稳定
- 想做标准页面、CRUD、表单、表格、抽屉、列表等模式化开发
- 想做复杂功能、跨文件变更、核心逻辑重构
- 想让 AI 参与联调、测试、Review、文档整理
- 想判断某个任务该用“直接改”还是“Spec 驱动”
- 想做多 Agent 分工协作

## 先做任务分级

收到开发请求后，先把任务归到以下三级之一：

### 1. 小颗粒需求：直接对话修改

适用：
- 改文案
- 改样式间距
- 小范围逻辑修正
- 显隐控制
- 单文件的小 bug

策略：
- 直接分析代码
- 直接改
- 给最小必要验证

不要为这种任务写重型 spec。

### 2. 中颗粒、标准化需求：规则/模板驱动生成

适用：
- 标准 CRUD 页面
- 标准表单页
- 标准列表页
- 业务组件脚手架
- 接口类型、service、hooks、页面骨架生成

策略：
- 先识别模式
- 读取项目里的规则、模板、示例
- 参照示例生成
- 保持目录结构、命名、接口封装、状态管理一致

这类任务重点不是“深度设计”，而是**对齐已有规范和样例**。

如需模板和产物格式，读：`references/patterns-and-outputs.md`

### 3. 中大颗粒复杂需求：Spec 驱动深度协作

适用：
- 新模块从 0 到 1
- 带复杂业务逻辑的功能
- 跨多个目录/层次的功能
- 核心流程重构
- 需要多步验收和阶段性交付的任务
- 高风险修改

策略：
- 必须先写 proposal / design / tasks
- 先明确 Why、边界、约束、验收标准
- 再拆执行步骤
- 每步都附验证方式

如需 spec 模板，读：`references/spec-templates.md`

## 标准工作流

### 模式 A：直接改

适用于小需求。

执行顺序：
1. 读取相关文件
2. 理解现状
3. 最小修改
4. 本地验证
5. 汇报变更与风险

输出至少要包含：
- 改了什么
- 为什么这样改
- 怎么验证

### 模式 B：规则/样例驱动生成

适用于标准化需求。

执行顺序：
1. 识别当前需求属于哪类页面/组件/模块
2. 读取项目规则文件、模板目录、参考实现
3. 提取必须遵守的结构约束
4. 生成目标代码
5. 运行格式化、类型检查、必要测试
6. 若首次输出不符合规范，优先重构到规范，而不是堆补丁

重点检查：
- 目录分层是否一致
- 命名是否一致
- 类型与接口封装是否一致
- 样式方案是否一致
- 是否复用了已有模式，而不是发明新模式

### 模式 C：Spec 驱动开发

适用于复杂需求。

标准阶段如下：

#### 1) Proposal
先定义：
- 为什么做
- 当前问题是什么
- 目标状态是什么
- 不做什么
- 风险点在哪里

#### 2) Design
定义实现方案：
- 模块边界
- 数据流
- 状态流
- 接口影响
- 依赖影响
- 迁移/兼容方案
- 风险与回滚点

#### 3) Spec
把用户可见行为写清楚：
- 页面/组件行为
- 交互细节
- 异常分支
- 空态、加载态、错误态
- 权限和边界条件

#### 4) Tasks
把实现拆成可执行步骤。
每个任务应包含：
- 输入
- 动作
- 输出
- 验收标准
- 验证命令或验证方法

#### 5) Execute
按 tasks 顺序执行，不要跳步骤。
高风险任务应分批提交和验证。

#### 6) Verify
至少做：
- 类型检查
- lint/格式检查
- 关键路径冒烟
- 需求验收核对

## 三层规范体系

为了让 AI 输出稳定，优先建立三层规范：

### 第一层：约束层
作用：告诉 AI “禁止什么、必须怎样”。

常见内容：
- TypeScript 约束
- 命名规范
- 注释规范
- 页面分层规范
- 服务层封装规范
- 样式规范
- 状态管理规范
- 测试规范

这层解决“别乱来”。

### 第二层：示范层
作用：告诉 AI “标准产出长什么样”。

常见内容：
- 列表页模板
- 表单页模板
- 编辑弹窗/抽屉模板
- service + types + hooks + page 的标准组合
- 通用组件示例
- 测试示例

这层解决“怎么写得像团队代码”。

### 第三层：视觉/产品上下文层
作用：告诉 AI “最终要长成什么样”。

常见内容：
- HTML 原型
- 设计稿导出
- PRD
- 交互说明
- 页面截图
- 云文档/知识库说明

这层解决“为什么这样做”和“用户看到什么”。

## 外部上下文接入原则

AI 很容易败在信息断层，不是败在写代码本身。

优先接入这些外部信息源：
- 接口文档
- PRD / 设计文档
- 页面原型
- 测试用例
- 发布/部署说明
- 日志与监控

原则：
- 能直连就直连，不要让用户重复复制粘贴
- 能结构化就结构化，不要只给截图
- 外部上下文进入后，要先整理成可执行约束，再开始改代码

## AI 失效的三种模式

### 1. 规范真空
表现：
- 功能能跑
- 但结构、风格、命名不符合团队习惯

应对：
- 把这次偏差补进规则或示范中
- 不要只修这一处，应该顺手沉淀规范

### 2. 信息孤岛
表现：
- 本地正常，CI 挂
- AI 每次分析都对，但只对当前暴露层
- 问题出在环境、平台、依赖安装、副作用配置

应对：
- 显式收集环境差异
- 比对本地、CI、测试、生产配置
- 把排障发现补成环境规范
- 不要假设 AI 自己知道企业内部环境

### 3. 目标模糊
表现：
- 用户说“优化一下”
- AI 直接开改
- 结果改偏

应对：
- 强制先澄清目标或先写 proposal
- 如果目标不清晰，先定义 success criteria，再执行

## 重构任务的特殊要求

重构和新功能不是一回事。

重构时必须额外写清：
- 哪些行为绝不能变
- 哪些模块可以迁移
- 哪些 API/props/导出名必须兼容
- 修改顺序是什么
- 每一步如何回归验证

不要一句“帮我重构一下”就直接大改。

重构的 tasks 应尽量细化到：
- 先确认现状
- 再迁移目录/职责
- 再更新引用
- 再类型检查
- 再冒烟

## 复杂排障的处理方式

当问题具备这些特征时，不要高估 AI：
- 只能在远端环境复现
- 一次验证成本很高
- 多根因叠加
- 根因藏在依赖源码或隐式安装行为里
- 日志不完整

这类问题要切换成“实验驱动排障”：
- 明确当前假设
- 一次只验证一个变量
- 记录每轮实验结论
- 尽量缩小环境差异
- 必要时读依赖源码

不要同时改 5 个地方再去赌 CI。

如需排障记录模板，读：`references/troubleshooting-playbook.md`

## 多 Agent 协作建议

任务足够大时，可以拆角色并行，但必须有主控。

推荐角色：
- **Planner**：需求澄清、proposal、design、任务拆解
- **Builder**：按任务实现代码
- **Reviewer**：检查规范一致性、边界条件、技术债
- **Verifier**：执行测试、类型检查、冒烟、截图对比
- **Documenter**：整理变更说明、迁移说明、验收记录

规则：
- 主控 Agent 只维护计划、边界、验收口径
- 实现 Agent 不擅自改变范围
- 审查 Agent 关注“是否符合 spec”，不重新发明方案
- 验证 Agent 产出客观结果，不给模糊判断

## 输出要求

默认输出应尽量包含：
- 任务分级判断
- 采用的协作模式
- 如需 spec，给出 proposal/design/tasks
- 关键约束
- 验证方式
- 风险点

复杂任务下，优先给结构化输出，不要直接一头扎进代码。

## 使用 references 的时机

根据任务类型按需读取：

- `references/spec-templates.md`
  - 当任务需要 proposal/design/spec/tasks 模板时读取
- `references/patterns-and-outputs.md`
  - 当任务是标准页面、组件、CRUD、生成式开发时读取
- `references/troubleshooting-playbook.md`
  - 当任务是 CI、构建、依赖、环境、远端排障时读取
- `references/multi-agent-playbook.md`
  - 当任务适合多 Agent 分工时读取

## 默认执行原则

1. 先判断颗粒度，再选流程
2. 先读规范和示例，再生成代码
3. 复杂任务先写 spec，再执行
4. 高风险修改分步落地、步步验证
5. 每次踩坑都尽量沉淀成规则、示例或排障手册
6. 不把 AI 当“会自己补业务常识的人”
7. 人负责边界、规范、取舍；AI 负责在确定空间里高速执行
