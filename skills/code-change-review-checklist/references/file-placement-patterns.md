# File Placement Patterns

## 目的

这个参考文件用来帮助判断“代码该放哪”。

## 一、先看逻辑属于哪一层

常见分层：

- `pages` / `views`：页面编排、数据拼装、交互串联
- `components`：可复用 UI 片段
- `hooks`：状态、副作用、行为封装
- `services` / `api`：请求、接口封装、数据获取
- `types`：类型定义
- `constants`：稳定常量、枚举映射
- `utils` / `helpers`：无业务状态的纯工具
- `stores` / `models`：全局状态、领域状态
- `adapters` / `transformers`：接口数据转换、视图模型转换

## 二、典型落点判断

### 1. 页面交互逻辑越来越多
信号：页面文件中出现大量事件处理、副作用、状态同步。

建议：
- 可复用或可独立理解的交互逻辑，拆到 `hooks`
- 请求逻辑拆到 `services`
- 数据转换拆到 `adapters` 或本地 helper

### 2. 同类 UI 结构重复出现
信号：多个页面都有相似块，只是字段和文案不同。

建议：
- 提炼为 `components`
- 不要把“只在一个页面存在的局部结构”过早升为全局组件

### 3. 接口返回需要大量加工
信号：页面里出现很多字段兜底、拼接、格式化。

建议：
- 把转换逻辑从页面中拿出来
- 放到 `adapter` / `transformer` / `mapper` 一类位置

### 4. 到处出现魔法值
建议：
- 多处共用且有业务语义的，放 `constants`
- 跟环境相关的，进配置
- 跟展示语言相关的，进 i18n

### 5. utils 文件持续膨胀
建议：
- 按领域拆小
- 避免把业务函数伪装成通用工具塞进 `utils`

## 三、快速判断法

如果一段代码：

- 依赖 UI 生命周期，通常不该进 `utils`
- 依赖接口协议，通常不该留在页面里
- 只做纯计算且无副作用，通常适合 helper / util
- 带业务语义且会复用，通常适合独立命名模块
- 一旦抽出来名字都起不清楚，说明可能还不该抽
