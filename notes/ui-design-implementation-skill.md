```md
# UI 设计与实现 Skill

把模糊的“做一个好看页面”收敛成稳定、可实现、可验收的规则。

## 适用目标

用于输出以下内容之一：

- 页面/模块级 UI 设计原则
- 设计实现约束清单
- 设计评审标准
- 前端落地规范
- AI 生成页面时的提示词/规则集

如果用户只给了一段设计理念，先将其整理为“流程 + 约束 + 检查项”的结构，而不是原样堆砌。

如果用户明确要求其中某一种成品，按下面方式处理：

- 要“规范版 / 团队规范 / 文档版”时，优先读取 `references/specification.md`
- 要“提示词版 / prompt / 给 AI 用”时，优先读取 `references/prompt-template.md`
- 如果既要规范版又要提示词版，两个参考文件都可直接复用

## 核心方法

始终按下面顺序组织内容。

### 1. 先定页面核心任务

先回答三个问题：

- 这个页面最核心的任务是什么？
- 用户进入页面后第一眼必须看到什么？
- 用户接下来最应该完成的唯一动作是什么？

要求：

- 一个页面只服务一个核心任务
- 不把登录、注册、设置、列表、详情、广告等多个目标混在同一层
- 主按钮只能有一个视觉主导项，其余都是次级操作

### 2. 先做空间布局，再做样式

先分配页面空间，再谈颜色和视觉。

布局要求：

- 视觉主体必须匹配需求核心
- 主体区域占据页面主要展示面积，通常不少于 2/3
- 辅助信息与辅助操作放到边缘区、顶部、底部或侧边，不得抢主体权重
- 页面结构优先清晰：内容区、操作区、分隔区

先问自己：

- 用户是否能在 1 秒内看懂页面用途？
- 主体是否明显强于辅助元素？
- 页面是否存在多个“视觉中心”？如果有，继续收敛

### 3. 设计信息层级，而不是堆信息

按“核心信息优先，详情延后”的原则组织内容。

要求：

- 最重要的信息直接展示
- 次要说明、扩展信息、低频信息放入弹窗、抽屉、折叠区、二级页
- 页面默认态不臃肿，不一次性塞满所有说明和入口
- 同层内容应有明确主次，不要平均用力

当信息很多时，优先采用：

- 摘要 + 查看详情
- 列表 + 详情页
- 主卡片 + 次级面板
- 默认收敛 + 按需展开

### 4. 统一排版比例与间距系统

不要凭感觉放元素，必须统一比例。

要求：

- 间距使用统一层级，不出现忽大忽小
- 字号有清晰梯度：标题、正文、辅助说明各自固定角色
- 控件尺寸统一，不要同页出现多套按钮高度、输入框高度、圆角尺寸
- 长文本默认左对齐，禁止大段居中排版

最低标准：

- 正文字号不低于系统可读下限
- 文字完整可读，不裁切、不变形、不旋转
- 区块之间保留呼吸感，避免操作区和信息区粘连

### 5. 操作元素必须克制

操作类 UI 只保留完成任务所必需的部分。

要求：

- 输入框、按钮、筛选器、标签等强操作元素不能密集堆叠
- 不同时突出多个 CTA
- 不制造“到处都能点，但不知道先点哪”的界面
- 用户路径应尽量单一、自然、连续

优先减少这些问题：

- 选择过载
- 操作干扰阅读
- 多入口竞争
- 视觉焦点被按钮区劫持

### 6. 全局只允许一套视觉语言

这是最容易失控的地方，必须显式约束。

要求：

- 全页面只允许 1 套主色 + 1 套辅助色 + 中性灰
- 不使用随机渐变、杂色、炫光或风格冲突材质
- 圆角、阴影、描边、分割线粗细统一
- 同一页面禁止混用拟物、扁平、玻璃态等多种风格

判断标准：

- 用户是否能感受到页面来自同一个设计系统？
- 任意两个卡片、按钮、弹窗是否像同一家产品做的？

### 7. 内容优先，装饰归零

所有视觉元素都必须有功能意义。

禁止：

- 无意义装饰线条
- 无功能色块
- 纯装饰图标
- 复杂背景纹理
- 为了“显高级”增加的发光、渐变、悬浮装饰

只保留三类元素：

- 内容区
- 操作区
- 分隔区

如果某个元素删掉后不影响理解和操作，就应该删。

### 8. 设计必须可实现

输出的 UI 必须能被真实组件系统落地。

只使用真实可实现组件思维：

- 按钮
- 输入框
- 卡片
- 列表
- 标签
- 导航
- 弹窗
- 抽屉
- 表单

禁止：

- 反人类布局
- 悬浮遮挡主要信息
- 文字被覆盖
- 假按钮
- 难以点击的交互热区
- 不规则异形图片裁切

图片/媒体区域要求：

- 使用标准比例容器预留空间
- 保证裁切规则一致
- 不让图片承担文字可读性背景

### 9. 可读性是硬指标，不是审美选项

必须满足基本可读性和可访问性。

要求：

- 文本与背景对比度至少满足 4.5:1
- 文字不能压在复杂背景、图片或强渐变上
- 长文本必须左对齐
- 信息区域优先纯净背景，减少识别干扰

评审时重点看：

- 是否一眼能扫出主标题、核心数据、主操作
- 是否存在灰字过浅、字号过小、行距过挤
- 是否有“看起来很设计，但读起来很累”的问题

### 10. 最终检查：页面是否只有一个明确答案

设计完成后，用下面的问题做收口：

- 这个页面的核心任务是否唯一？
- 用户是否知道第一步该做什么？
- 是否只有一个主按钮被突出？
- 是否所有非必要元素都已删除？
- 是否还能继续减少信息和操作噪音？

如果用户会问“我到底该点哪里”，说明设计还没完成。

如果用户只要短版总结，可压缩为下面这组规则：
```

## specification.md

```md
# UI 设计实现规范版

## 1. 目标

本规范用于统一页面级 UI 的设计与实现标准，保证界面满足以下要求：

- 一眼可理解页面用途
- 主次分明，信息不过载
- 风格统一，可持续扩展
- 可被真实前端组件稳定实现
- 可读、可用、可评审、可验收

---

## 2. 总原则

### 2.1 单页面任务唯一

一个页面只允许承载一个核心任务。

要求：

- 页面核心目标必须唯一
- 禁止把登录、注册、设置、列表、详情、广告等多个任务并列塞进同一层
- 同级操作中只允许一个主操作按钮突出显示
- 用户进入页面后，应当立即知道“这是干什么的”和“下一步该点哪里”

### 2.2 内容优先于装饰

所有视觉元素必须服务于信息表达和任务完成。

要求：

- 无意义装饰元素一律去除
- 不为了“高级感”添加额外图形、线条、色块、发光、纹理
- 页面仅保留内容区、操作区、分隔区三类必要结构

### 2.3 可实现性优先于概念表达

设计方案必须能被真实组件系统稳定落地。

要求：

- 所有控件必须对应真实可实现组件
- 禁止假按钮、不可点击热区、遮挡文字、反人类布局
- 图片、卡片、弹窗、表单、列表等都必须符合常规前端实现逻辑

---

## 3. 页面设计流程

### 3.1 先规划空间布局与视觉主体

先确定页面空间分配，再处理颜色和样式。

要求：

- 视觉主体必须与页面核心需求一致
- 视觉主体应占据页面主要展示区域，通常不少于 2/3
- 页面不能同时存在多个同等级视觉中心
- 辅助信息和辅助操作应布置在边缘区域，不得抢占主体视觉权重

评审标准：

- 用户是否能在 1 秒内理解页面用途
- 主体区域是否足够突出
- 是否存在视觉焦点冲突

### 3.2 再确定辅助功能 UI

辅助元素只承担解释、补充和辅助操作作用。

包括但不限于：

- 标题
- Tag
- 提示信息
- 次级按钮
- 面包屑
- 辅助操作区

要求：

- 统一放置在顶部、底部、边缘或次级区域
- 不得与主体内容争夺第一视觉层级
- 不得因为辅助元素过多导致页面显得拥挤

### 3.3 设计主体信息展示方案

根据信息重要性做展示分层。

要求：

- 核心信息必须直接展示
- 次级信息应弱化展示
- 详情信息、补充说明、低频功能应进入弹窗、抽屉、折叠区或二级页面
- 不允许把所有信息平铺到同一层级

推荐模式：

- 摘要 + 详情
- 列表 + 详情
- 主信息卡 + 次级说明区
- 默认收起 + 按需展开

---

## 4. 排版与比例规范

### 4.1 间距规范

要求：

- 页面外边距、模块间距、组件间距必须使用统一层级
- 不允许同一页面出现无规律的大间距和小间距混用
- 强操作区与信息区之间必须保留足够呼吸感

### 4.2 字体规范

要求：

- 建立稳定字号层级：标题、二级标题、正文、辅助文字
- 正文字号不得低于系统可读下限
- 长文本必须左对齐
- 禁止大段正文居中排版

### 4.3 控件尺寸规范

要求：

- 同页面按钮高度统一
- 输入框高度统一
- 卡片圆角统一
- 分隔线粗细统一
- 阴影风格统一

不允许：

- 一个页面出现多种按钮高度
- 同层级卡片使用不同圆角系统
- 同类控件视觉重量不一致

---

## 5. 操作设计规范

### 5.1 操作入口必须精简

要求：

- 页面操作入口数量应被严格控制
- 输入框、按钮、筛选器、标签等不得密集堆叠
- 避免形成选择过载
- 同一层只突出一个主按钮，其余使用次级按钮或文字按钮

### 5.2 用户路径必须单一清晰

要求：

- 页面必须有明确的完成路径
- 用户不应在多个并列入口之间犹豫
- 操作流程应自然连续，不应频繁跳转注意力

禁止：

- 多个主 CTA 并列竞争
- 阅读区被操作区压缩
- 操作密度高到影响信息理解

---

## 6. 视觉语言统一规范

### 6.1 色彩系统

要求：

- 全页面只允许 1 套主色、1 套辅助色和中性灰系统
- 禁止随机渐变、杂色、炫光、冲突材质
- 色彩只用于表达层级、状态和主次，不用于装饰堆砌

### 6.2 组件风格统一

要求：

- 圆角规则全局统一
- 阴影规则全局统一
- 描边与分割线规则全局统一
- 同一页面只能使用一套设计风格

禁止：

- 扁平风与玻璃态混用
- 拟物风与极简风混用
- 同一页面出现多套卡片风格

---

## 7. 内容与装饰约束

### 7.1 装饰元素最小化

禁止添加以下无功能元素：

- 装饰线条
- 装饰性色块
- 无意义图标
- 复杂背景纹理
- 与内容无关的视觉点缀

判断标准：

- 删除该元素后，若不影响理解和操作，则该元素不应保留

### 7.2 图片与媒体规范

要求：

- 图片区域必须预留标准比例容器
- 图片裁切规则必须一致
- 不允许异形裁切
- 不允许文字压在复杂图片背景上

---

## 8. 交互实现约束

要求：

- 所有控件都必须可点击、可输入、可聚焦、可反馈
- 不允许出现视觉上像控件、实际不可用的伪交互
- 不允许悬浮元素遮挡主要文本和主操作
- 不允许出现点击热区过小或相互重叠的情况
- 文本必须完整显示，不允许裁切、旋转或艺术化变形

---

## 9. 可读性强制标准

要求：

- 文字与背景对比度至少满足 4.5:1
- 正文不得压在图片、复杂背景、强渐变上
- 正文区域优先使用纯净背景
- 长文本必须左对齐
- 行距、段距要保证扫读效率

评审重点：

- 主标题是否醒目
- 核心信息是否能被快速定位
- 字号是否过小
- 灰度是否过浅
- 阅读是否费力

---

## 10. 最终验收清单

在设计交付或前端实现完成后，必须逐项检查：

- 页面是否只服务一个核心任务
- 用户是否一眼能理解页面用途
- 主体区域是否占据主要视觉面积
- 辅助元素是否退居边缘
- 信息是否已做主次分层
- 是否仅保留一个主按钮
- 操作入口是否足够克制
- 间距、字号、圆角、阴影是否统一
- 全页面是否只有一套视觉语言
- 是否删除了所有无功能装饰
- 所有控件是否真实可实现
- 文本是否完整、清晰、可读
- 用户是否明确知道下一步操作

如果以上任一问题回答为“否”，则说明设计尚未完成收敛。
```

```md
# UI 设计实现提示词版

你是一个偏产品化、可落地的 UI 设计助手。你的任务不是堆视觉效果，而是把页面做得清晰、统一、可实现、可操作。

请严格遵守以下规则：

## 一、页面目标

- 一个页面只允许一个核心任务
- 用户进入页面后必须一眼知道：这个页面是干什么的
- 用户必须立刻知道下一步该点哪里
- 同级操作中只允许一个主按钮突出，其余全部次级化

## 二、布局优先级

- 先规划空间布局，再设计样式
- 视觉主体必须和核心任务一致
- 视觉主体占据至少 2/3 的主要展示区域
- 辅助信息、标题、Tag、说明、次级按钮统一放边缘或次级区域
- 不允许一个页面出现多个视觉中心

## 三、信息展示规则

- 核心信息直接展示
- 次级信息弱化展示
- 详情、补充说明、低频功能放到弹窗、抽屉、折叠区或二级页面
- 禁止把所有内容堆在一个平面里
- 页面默认态必须清爽，不臃肿

## 四、排版规则

- 间距必须统一，不能忽大忽小
- 字号必须有层级：标题、正文、辅助说明分工明确
- 正文最小字号不得低于系统可读下限
- 长文本一律左对齐
- 禁止大段文字居中
- 控件尺寸统一：按钮高度、输入框高度、圆角、阴影、描边必须统一

## 五、操作设计规则

- 输入框、按钮、筛选器等操作元素不能密集堆叠
- 页面不提供过多操作入口，避免选择过载
- 不允许多个主 CTA 同时抢注意力
- 用户路径必须单一、自然、连续
- 不允许出现“不知道该先点哪里”的界面

## 六、视觉风格规则

- 全页面只允许一套视觉语言
- 只允许 1 套主色 + 1 套辅助色 + 中性灰
- 禁止随机渐变、杂色、炫光、风格冲突材质
- 圆角、阴影、描边、分割线粗细必须统一
- 不允许混用拟物、扁平、玻璃态等风格

## 七、装饰限制

- 所有视觉元素必须服务内容和操作
- 删除所有无意义装饰
- 禁止纯装饰线条、色块、背景纹理、无功能图标
- 只保留：内容区、操作区、分隔区
- 如果删掉某个元素不影响理解和操作，就删掉它

## 八、实现约束

- 所有控件必须是真实可实现组件
- 只能使用常规 UI 组件思维：按钮、输入框、卡片、列表、标签、导航、弹窗、抽屉、表单
- 禁止假按钮、不可点击热区、悬浮遮挡、文字压盖
- 文本必须完整可读，不允许裁切、旋转、艺术化变形
- 图片区域必须使用标准比例容器，不允许异形裁切

## 九、可读性标准

- 文字与背景对比度必须 ≥ 4.5:1
- 文字不能压在复杂背景、图片或强渐变上
- 正文区域优先纯净背景
- 信息必须能被快速扫读
- 禁止“看起来很炫但读起来很累”的设计
```
