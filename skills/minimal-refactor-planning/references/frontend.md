# 前端改造参考

当问题主要集中在 UI 代码、页面编排、状态归属、组件职责或交互流程时，读取这个 reference。

## 常见前端异味

### 1. 多个组件各自实现同一套流程
典型信号：

- 多个按钮 / 菜单 / 空态都在触发同一条异步链路
- 每个组件里都维护了一份相似但不完全相同的逻辑

优先方向：

- 把真正的处理流程收进一个共享 owner
- UI 入口只保留触发能力

### 2. 展示和副作用混在一起
典型信号：

- 一个组件既负责渲染 UI，又负责文件处理、请求、数据整形、成功提示、重试等

优先方向：

- 渲染尽量留在组件里
- 把流程编排抽到 hook / controller / shared module

### 3. 状态归属不清晰
典型信号：

- 一个状态同时影响多个同级组件
- 回调在多层之间来回传，但没有明确 owner
- 子组件实际上在控制其它不相关组件的行为

优先方向：

- 把状态提升到最近的共同父层
- 向下只传必要状态和显式 handler

### 4. 间接通信过多
典型信号：

- CustomEvent
- window 事件
- 临时全局信号
- ref 被拿来当隐藏命令通道

优先方向：

- 显式 props 回调
- 父层持有共享 hook
- 稳定的 controller / service 边界

只有真的跨树、跨运行时、跨容器时，才保留事件机制。

### 5. 状态提升过头
典型信号：

- 本来只是页面局部行为，却被放进 page/global store

优先方向：

- 只提升到真正需要协调的那一层
- 不要为了“统一”就默认全局化

## 如何选抽象

### shared helper
适合抽出纯逻辑：

- parse
- transform
- map
- validate
- payload 生成

### hook
适合带交互状态的流程：

- pending 状态
- 选择状态
- 重试
- 错误/成功反馈
- 需要命令式触发的 UI 行为

### page-level owner
适合多个同级组件共享一套能力或状态。

### service / uploader / manager
适合异步编排、远程调用、文件/资源处理，不适合继续塞在视觉组件里。

## 前端问题回答时应说清楚什么

一个好的前端改造方案，通常要明确：

1. 哪一层持有状态
2. 哪个组件变成 trigger-only
3. 哪些逻辑被统一收口
4. 哪种通信机制要被移除
5. 哪条回归路径必须重测

## 常用表述

- “这样可以让 UI 入口保持薄，只把编排逻辑收进一个 owner。”
- “空态和菜单会变成同一条流程的两个入口。”
- “这能把事件式通信改成显式的父层持有。”
- “这次先拆到这里就够了，再细拆属于提前设计。”

## 常见坑

默认不要这样建议：

- 为页面局部行为上全局状态
- 边界还没稳定就抽很多小 hook
- 让多个组件继续各自维护同一套异步逻辑
- 用事件机制解决本该用层级关系解决的问题
- 为一次性流程硬做通用框架抽象
