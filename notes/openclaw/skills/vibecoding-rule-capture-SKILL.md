# Vibecoding Rule Capture

## Overview

这个 skill 用来把“AI 这次写错了”变成“AI 下次少犯同类错误”。

核心目标：**当一次失误已经暴露出稳定模式时，把它提炼成短小、可执行、可长期复用的规则，追加到 AGENTS.md，而不是只在当前对话里口头记住。**

## 适用场景

当出现以下任一情况时，触发本 skill：

- 用户明确要求创建、补充、整理、沉淀、更新 AGENTS.md 规则
- 用户说出类似这些话：
  - “以后别再这样了”
  - “这个规则记一下”
  - “把这个坑写进 AGENTS.md”
  - “你老是在这里犯错”
  - “把这个经验沉淀下来”
  - “做成规范”
  - “做个 skill，让你出错就记规则”
- AI 刚刚完成一次返工，且返工原因不是业务变更，而是方法错误、流程错误、落点错误、验证缺失、沟通缺失、结构判断错误
- AI 在同一会话内连续两次出现同类错误
- 用户指出“这不是第一次了”“上次也这样”“你总是这样写”
- 一次错误可以被抽象为稳定规则，而不是一次性特例
- 在 vibecoding、直接改代码、Spec 驱动开发、review、自查、排障、联调后，需要做一次“错误 -> 规则”沉淀
- 完成代码修改后，发现问题本质上是缺少明确 guardrail，而不是单次粗心

## 不适用场景

以下情况不要写进 AGENTS.md：

- 纯粹一次性的业务决定，没有复用价值
- 只针对某个临时文件、临时分支、临时实验的偶发情况
- 用户表达的是偏好，但还没有稳定到值得固化为长期规则
- 问题并不是“缺规则”，而是这次单纯执行疏忽
- 规则表述只能针对一个极窄场景，离开当前上下文就无法理解
- 需要写入的是项目知识、接口文档、产品约束、模块说明，这类内容更应该放到项目文档、skill reference、README 或专门规范文件里

## 触发条件设计原则

目标不是“任何错误都写规则”，而是识别**值得长期固化的错误模式**。

优先捕捉以下类型的可复用错误：

### 1. 流程类错误
例如：
- 没先读关键文件就开改
- 没先看现有模式就自创实现
- 没做必要验证就宣称完成
- 没确认写入目标就直接覆盖
- 没先判断该不该改 AGENTS.md / MEMORY.md / skill 文件

这类最适合写规则，因为它们高复用、跨任务稳定存在。

### 2. 落点类错误
例如：
- 代码放错目录
- 把业务逻辑塞进错误层级
- 本应拆文件却继续往大文件里塞
- 把临时规则写进长期规范
- 把项目级约束写进用户级全局文件

### 3. 验证类错误
例如：
- 没跑测试 / lint / typecheck 就汇报完成
- 没检查文件是否已存在就覆盖
- 没验证目标环境、路径、权限、输出结果
- 没做最小冒烟验证

### 4. 沟通类错误
例如：
- 在需要确认时没确认
- 改动超出用户明确授权范围
- 未区分“建议”与“已执行”
- 复杂任务没先给方案就直接大改

### 5. 结构与质量类错误
例如：
- 重复造轮子而没复用现有模式
- 为了快把实现糊到不合适位置
- 引入明显硬编码
- 忽略边界情况
- 用补丁叠补丁，没有顺手收敛结构

### 6. 记忆与沉淀类错误
例如：
- 明知应该长期记住却没写入文件
- 发现重复坑后没有把经验转为规则
- 口头说“下次注意”，但没形成可执行约束

## 决策流程

当你怀疑“这次错误应该沉淀成规则”时，按下面流程处理。

### 第一步：确认这是不是“模式化错误”
先问自己：

1. 这次错误是否可能再次发生？
2. 它是否跨任务、跨文件、跨项目仍然成立？
3. 一条短规则是否能显著降低未来复发概率？
4. 这是流程问题/决策问题，而不是纯手滑？

如果以上大多为“是”，进入下一步。

### 第二步：判断规则应写到哪里
优先判断目标文件，不要默认都写 AGENTS.md。

- **写 AGENTS.md**：
  - 这是当前 workspace 的长期协作规则
  - 对未来大量任务都有帮助
  - 属于 agent 行为约束、工作习惯、工程协作守则

- **不要写 AGENTS.md，改写别处**：
  - 项目实现细节 -> 项目文档 / reference
  - 用户个人长期信息 -> MEMORY.md 或 daily memory
  - 具体技能用法 -> 某个 skill 的 SKILL.md / references
  - 某个模块规则 -> 模块内文档或代码注释

### 第三步：把错误转成“可执行规则”
规则必须满足：

- 简短
- 明确
- 可执行
- 可检查
- 不依赖当前对话上下文

坏例子：
- “以后认真一点”
- “别搞错了”
- “注意这个问题”

好例子：
- “在覆盖 notespace 文件前，先确认目标路径和完整正文，再执行写入与提交。”
- “修改现有页面前，先找一个同类页面对齐目录结构、状态组织和接口封装，不要凭空发明新模式。”
- “完成代码修改后，至少做一次与本次变更强相关的最小验证；未验证时不得表述为‘已完成’。”

### 第四步：检查是否会污染 AGENTS.md
如果拟加入的规则具有以下问题，不要写：

- 太长
- 太具体
- 只能服务一次任务
- 和现有规则重复
- 实际是某个具体案例，不是抽象后的原则
- 写进去后会让 AGENTS.md 越来越像流水账

### 第五步：以最小追加方式写入
默认采用**append-only**思路：

- 不大改 AGENTS.md 结构
- 不重写整份文件
- 只在合适的小节下追加简短规则
- 若没有合适小节，再新增一个简洁小节

## 规则写作标准

每条新增规则尽量满足以下格式之一。

### 格式 A：直接命令式
适合强约束。

- “在修改跨文件功能前，先确认每个改动落点是否符合现有目录职责。”
- “当任务要求写入外部仓库时，先确认目标路径与文件内容，再执行写入、提交、推送。”

### 格式 B：触发条件 + 动作
适合条件明确的场景。

- “当用户要求上传笔记到 notespace 时，先确认目标文件名和最终 Markdown 正文，再写入并提交。”
- “当本次修改涉及现有模式复用时，先读取同类实现，再决定是否新建结构。”

### 格式 C：禁止式
适合高危坏习惯。

- “不要在未做最小验证时宣称任务已完成。”
- “不要把项目级一次性实现细节写入 AGENTS.md。”

## 推荐的小节归类

往 AGENTS.md 追加规则时，优先归入现有语义最接近的小节。没有时再新增。

常见归类：

- `## 开发流程`
- `## 修改前检查`
- `## 修改后验证`
- `## 文件落点与结构`
- `## 沟通与确认`
- `## 记忆与沉淀`
- `## 禁止事项`

## 默认执行行为

当本 skill 被触发时，默认按以下顺序工作：

1. 回顾刚才的错误或返工原因
2. 提炼成一句可复用规则
3. 判断是否真的该写入 AGENTS.md
4. 找到 AGENTS.md 中最合适的小节
5. 以最小改动追加规则
6. 向用户汇报：
   - 识别到的错误模式是什么
   - 新增了哪条规则
   - 加在 AGENTS.md 哪个位置

## 输出模板

### 简版
适合对话内快速沉淀：

- **错误模式**：一句话
- **新增规则**：一句话
- **写入位置**：AGENTS.md 的哪个小节

### 详细版
适合用户明确要求复盘：

- 这次错误属于哪一类
- 为什么值得写成长期规则
- 规则草案
- 为什么写到 AGENTS.md 而不是别处
- 追加位置
- 以后这条规则会拦住哪些类似错误

## 和其他 skill 的关系

- 如果任务本身是“创建 / 改进一个 skill”，优先结合 `skill-creator`
- 如果错误来自代码落点、边界、硬编码、结构坏味道，可结合 `code-change-review-checklist`
- 如果错误来自开发协作方式、需求不清、Spec 缺失，可结合 `ai-spec-coding`

## 最重要的原则

**不要把每次失误都写进 AGENTS.md；只把“下次真的值得拦住的模式”写进去。**

AGENTS.md 应该越来越像高价值 guardrails，而不是错误日志。
