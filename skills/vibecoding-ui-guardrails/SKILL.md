# Vibecoding UI Guardrails

把零散的审美偏好收敛成稳定、可复用、可落地的界面约束。

核心目标不是“做得更花”，而是让界面在任何层级都保持：

- 克制
- 功能优先
- 高信息密度但可扫描
- 视觉层级服务理解与决策
- 可被真实组件系统稳定实现

当用户给出一批 UI 偏好、设计要求或评审意见时，不要原样堆砌。先把它们整理成**设计哲学 → 布局层级 → 信息呈现 → 视觉约束 → 交互规则 → 组件通用规则 → 最终检查**这条决策链。

## 适用场景

当用户有以下诉求时，使用本 skill：

- 想总结一套通用 UI 设计规范或 guardrails
- 想把 Vibecoding 的设计要求整理成 AI 可执行规则
- 想约束页面、组件、弹窗、列表、卡片、仪表盘的视觉与布局
- 想控制信息密度，避免页面臃肿、卡片泛滥、层级混乱
- 想做设计稿评审、前端落地约束、组件规范收敛
- 想统一视觉语言，减少风格漂移和装饰性设计
- 想把“好看”“克制”“高级感”转成能真正指导实现的规范

## 使用原则

收到与 UI 规范相关的请求后，默认按下面顺序组织输出，不要跳级。

1. 先定义设计哲学
2. 再定义布局与层级规则
3. 再定义信息呈现与辅助内容规则
4. 再定义视觉元素约束
5. 再定义交互与操作优先级
6. 再补组件级通用规则
7. 最后给出决策 checklist

如果用户只给了一堆偏好、批评或修改意见，先抽象出可复用原则，再输出规范。不要把一次性的具体措辞直接当规则。

## 1. 设计哲学

所有具体规则都必须服从下面四条。

### 功能优先于装饰

每一个新增元素都必须能证明它至少满足以下之一：

- 帮助用户理解当前状态
- 指导下一步操作
- 防止错误

无法满足任意一项的，默认不加。

适用对象包括但不限于：

- 标题
- 标签
- 说明文案
- 统计摘要
- 预览文案
- 容器
- 阴影
- 圆角
- 边框
- 徽标
- 色块

### 层级服务于阅读与决策

视觉层级不是为了“看起来完整”，而是为了帮助用户更快理解结构、找到重点、完成动作。

因此：

- 只有在存在明确功能分区时，才允许引入额外卡片、边框、容器、留白层
- 不为了制造完整感而堆叠纯装饰性容器
- 不为了强调而把所有元素都做成强调态

### 克制与一致性

默认使用最小必要视觉语言。

要求：

- 同一界面内圆角、阴影、边框的使用保持一致
- 同一组件内不要出现相互矛盾的边界表达
- 不要同时混用多套强调方式
- 不要在局部发明一套与全局不兼容的视觉语法

### 信息密度与可扫描性优先

优先追求“一眼可扫”的信息组织，而不是松散、分裂、到处解释。

默认动作：

- 能从上下文推断的内容删除
- 能合并的字段合并
- 能收敛的区块收敛
- 能延后显示的内容延后显示

## 2. 布局与层级

### 外层承载边界

在已有主内容容器内，不要再给核心内容额外套第二层纯装饰性容器。

优先让最外层布局负责：

- 留白
- 边界
- 呼吸感
- 主次分区

只有当内层容器承担明确功能时，才允许继续加层。

### 页面级与组件级要分开处理

不要把页面级规则直接套到组件级，也不要把组件级表达错误放大到页面级。

例如：

- 页面级适合处理主体分区、主视觉重心、导航关系
- 组件级适合处理字段排列、头部压缩、按钮优先级、元信息贴附

判断时先问：

- 这是页面问题，还是局部组件问题？
- 这个层级是否真的需要新的视觉边界？

### 结构性区域应匹配信息密度

导航、侧栏、过滤栏、统计区、摘要区等结构性区域的体量，必须跟内容密度匹配。

规则：

- 内容稀疏时优先压缩尺寸
- 折叠/展开入口必须明显
- 不要为了对称感保留过宽侧栏或过高头部
- 不要让辅助区域挤压主任务区域

### 空状态、统计区、摘要区优先收敛

同层信息默认优先合并为：

- 连续信息带
- 紧凑列表
- 有分隔的统一结构

而不是拆成多张彼此争抢注意力的卡片。

## 3. 信息呈现与辅助内容

### 辅助信息必须贴近所属对象

时间、状态、快捷键、说明、标签等元信息必须紧贴关联内容，不要散落在界面不同角落。

要求：

- 遵循一致对齐规则
- 与所属对象形成稳定阅读关系
- 不跨区域漂浮展示

### 低优先级元信息自然后置

创建时间、更新时间、场景说明、补充描述等低价值信息，默认放在：

- 内容之后
- 主操作之后
- 正文区尾部或弱化区域

不要把它们插在核心识别信息与主操作之间。

### Header 极简

卡片、列表项、面板、模块头部的 header 应只承载：

- 识别信息
- 主状态

标签、场景、时间、补充说明等次级信息，应下沉到正文区、尾部或折叠区。

### 重复元信息要降级

多个实例中高度重复、区分度低的元信息，不应长期占据主视觉位置。

默认降级方式：

- hover
- tooltip
- 详情页
- 折叠态
- 更多菜单

### 统计摘要区极致精炼

摘要区优先展示“标签 + 关键值”这类一眼可扫的信息。

原则：

- 能从上下文推断的解释句删除
- 能压成单行就不要拆段
- 能并排就不要堆多层说明

## 4. 视觉元素

### 圆角使用克制

默认使用系统级、克制的圆角，不放大圆角来制造“设计感”。

只有在以下情况才允许加重圆角：

- 明确提升层级表达
- 扩大点击热区感知
- 满足系统一致性

### 激活态边界要完整

对于圆角容器的选中态、激活态、聚焦态，必须采用完整边界系统。

可选方式：

- 全边描边
- 完整色条系统
- 完整阴影强化

避免只描单边、只亮一角、只改局部边界，导致视觉语言自相矛盾。

### 标题层级优先靠排版建立

页面标题、模块标题、区块标题，应优先依靠：

- 字号
- 字重
- 间距
- 对齐关系

建立层级。

不要叠加与标题语义重复的装饰性眉题、小标签、徽标条来硬做层级。

### 弹窗标题尺度匹配任务规模

弹窗标题默认使用区块级标题尺度，明显小于页面主标题。

规则：

- 单步输入任务：进一步收紧
- 确认任务：更简更短
- 不要把小弹窗做成“迷你页面”标题系统

## 5. 交互状态与操作

### 状态区分必须清晰

禁用态、进行中、完成态、警告态、不同类型内容的差异，不能只依赖轻微色差。

应使用可识别组合：

- 图标
- 字重
- 透明度
- 轮廓
- 背景对比

### 主 CTA 必须明确且位置稳定

卡片、列表、面板、弹窗中的主 CTA，通常应落在扫描路径末端，常见是底部右侧或主内容后的收口位。

要求：

- 主操作只保留一个实心重点
- 次级操作默认降级为 outline、ghost 或更多菜单
- 不要多个实心按钮同时抢主导权

### 低频和容错类操作默认收敛

低频、补充性、容错类操作，不要与主 CTA 并列占据核心视觉区。

默认收纳到：

- overflow menu
- context menu
- 二级操作区
- 折叠区域

### 右上角工具位用途固定

卡片、列表项、面板右上角默认只留给上下文工具操作，例如：

- 更多
- 展开
- 切换
- 固定
- 收起

被动展示信息如时间、标签、状态文本，不应占用这个位置。

## 6. 组件通用规则

### Header

页面头部、模块头部、面板头部默认优先收敛为单行操作栏。

当同时包含页面身份信息与返回/关闭/次级操作时：

- 左侧放标题与关键元信息
- 右侧放操作按钮
- 少量状态或分类信息贴近标题组织
- 不为了摆竖排元信息额外抬高整个 header

### Dialog / 弹窗

控制弹窗尺寸的优先顺序应为：

1. 内容单列化
2. 删除冗余说明
3. 启用内部滚动
4. 最后才考虑放大弹窗

补充规则：

- 标题已明确当前对象时，正文不要再重复对象摘要
- 单步表单或确认弹窗，不默认加入教程式说明
- 不要在弹窗里复述“创建后会怎样”这类低价值预告

### 表单与字段

字段旁辅助说明只有在**防止输入错误**时才应常驻。

其他情况默认降级到：

- tooltip
- help icon
- 按需展开说明

未准备好交付的模块默认隐藏，不要用骨架屏、假列表、空壳区域撑页面，除非这些占位内容能明确指导下一步操作。

### Dashboard / 卡片

仪表盘、摘要卡片、列表卡片的原则是：高信息密度，但可扫描。

优先策略：

- 相关字段压缩成单行“标签 + 值”
- 主视觉区域优先给识别信息、关键状态、唯一主动作
- 卡片底栏若承载动作，就只承载动作
- 创建时间等元信息不要混进动作底栏

## 7. 最终决策 checklist

在做任何设计决策前，用下面三个问题先过一遍。

1. 这个元素、容器、文案，是否真正帮助用户理解状态、指导操作或防止错误？
2. 如果删掉它，体验是否会明显变差？不会就删。
3. 当前视觉层级和信息密度是否已经刚刚好？再加一层是否会挤压核心内容？

## 标准输出方式

当用户要求把设计要求整理成规范、团队规则、提示词、评审标准时，优先输出为下面结构：

1. 设计哲学
2. 布局与层级
3. 信息呈现与辅助内容
4. 视觉元素约束
5. 交互状态与操作优先级
6. 组件通用规则
7. 最终自查清单

这样更适合后续继续转换成：

- 团队规范文档
- AI 提示词
- 设计评审 checklist
- 前端实现 guardrails

## 精简版原则

如果用户只要短版总结，可压缩为以下规则：

- 功能优先于装饰
- 层级只为阅读和决策服务
- 外层负责边界，内层少加纯装饰容器
- 信息能合并就合并，能后置就后置
- Header 只放识别信息和主状态
- 统计区和摘要区优先压成一眼可扫的结构
- 圆角、边框、阴影使用克制且一致
- 激活态必须有完整边界语言
- 主 CTA 只能有一个真正突出
- 低频操作默认收进更多菜单
- 弹窗优先收内容，不优先放大尺寸
- 表单说明只有在防错时才常驻
- 卡片与仪表盘优先高密度、可扫描
- 每个元素都要能解释自己为什么存在
