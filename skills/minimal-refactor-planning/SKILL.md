# Minimal Refactor Planning

当用户**不是要你立刻写代码**，而是希望你先给出一个低风险、可落地、改动受控的改造方案时，使用这个 skill。

## 何时使用

以下类型的请求适合触发：

- “先别写代码，先给方案”
- “给我最小改法”
- “比较一下这几种改法”
- “这块应该怎么拆”
- “帮我梳理下实现方案”
- “给个落地计划”
- “尽量少动现有代码，怎么改”

常见问题特征：

- 多种职责混在一起
- 多个入口重复实现同一套流程
- 状态归属不清楚
- 模块/组件之间依赖关系别扭
- 依赖临时拼接出来的通信方式（事件、胶水代码、隐式约定）
- 用户要的是明确建议，不是空泛架构讨论

## 目标

输出一个满足以下要求的方案：

1. 改动范围尽量小
2. 取舍清楚
3. 足够具体，下一步可以直接实现
4. 注意回归风险
5. 不做不必要的抽象

默认倾向：

- 优先选最小可行改法
- 优先显式依赖，不绕
- 优先把共享逻辑收口到一处
- 除非用户明确要求，否则不做平台级重构

## 默认工作方式

除非用户另有要求，按这个顺序处理：

1. 识别当前实现里真正混在一起的职责
2. 找出最小、最合适的承接边界
3. 给出 2～3 个可行方案
4. 明确推荐其中一个
5. 说明主流程和受影响范围
6. 给出小而够用的验证计划
7. **先不写代码**

## 默认输出结构

### 1. 先给结论

开头先直接说推荐哪种改法，并用一两句话说明原因。

好的表达方式例如：

- 我建议这次先做最小拆分：把共享流程收进一个统一 owner，入口只保留触发。
- 这次不建议继续细拆，一个共享抽象已经够了。

### 2. 方案对比

列 2～3 个方案即可。每个方案简要说明：

- 逻辑放在哪
- 谁持有核心状态 / 核心流程
- 依赖关系是变干净还是更绕
- 是否适合这次需求范围

不要把几个方案写得像都差不多。推荐方案应当很明显。

### 3. 推荐设计

说明：

- 要不要新增抽象；如果要，新增什么  
  （如 shared module、helper、service、manager、hook、controller、adapter）
- 它放在哪一层
- 它负责什么
- 它不负责什么
- 周边模块如何接入它

### 4. 主流程

按步骤说明主执行路径：

1. 从哪里开始
2. 由谁接住
3. 谁执行核心逻辑
4. 成功后如何回传
5. 失败后如何处理
6. 哪些已有行为保持不变

根据问题类型，也可以写成“数据流”“调用流”或“执行流”。

### 5. 改动范围

列出实际要动的东西：

- 哪些逻辑要迁移
- 哪些位置改成薄触发器 / 调用方
- 新增什么共享单元
- 哪些临时通信方式要移除
- 哪些接口 / 参数需要调整

### 6. 验证

验证范围要和改动规模匹配。通常只包括：

- 抽出的纯函数 / 辅助逻辑验证
- 定向测试
- 局部 lint / typecheck / build
- 关键回归路径

不要默认把范围扩大成全量重构或全量测试补齐。

## 决策准则

### 优先收口真正共享的流程

如果多个入口最终做的是同一件事，就把真正的处理逻辑集中到一处。边缘位置只保留：

- 触发
- 参数传递
- 展示状态或结果

避免在多个地方复制流程编排。

### 优先显式依赖

如果当前设计依赖这些东西：

- 全局事件
- 隐式约定
- 反向归属
- 分散的胶水代码
- 很难追踪的跨模块调用

优先改成：

- 显式注入
- 明确的方法 / 回调边界
- 一个共享 owner
- 更直的调用链

### 优先一个新抽象，而不是很多层

默认不要过度细拆。

只有在以下至少一条明显成立时，才继续加层：

- 逻辑已经被很多地方复用
- 行为明显会往多个方向演进
- 测试隔离确实需要独立边界
- 一个抽象已经开始承载多个不相关职责

### 优先保留现有语义

默认假设：这次重构要保持行为不变，除非用户明确要求改行为。

重点保护：

- 现有业务语义
- 成功 / 失败处理方式
- 回调时机
- 返回结构 / 输出预期
- 用户可见主流程

### 避免过度设计

如果某个方案“理论上更优雅，但这次不值”，要直接指出来。

常见警报：

- 新抽象太多
- 为未来扩展提前付出过多成本
- 为局部问题引入过重基础设施
- 把范围扩大到用户并没要求的层面

## 输出风格要求

- 先结论，后展开
- 工程判断感强，不要像写教材
- 重点讲取舍，不讲空话
- 默认站在低风险、快落地的角度
- 不要把决定权完全甩回给用户
- 用户没要求前，不要直接开始写代码

## 可复用回答骨架

默认按这个结构组织：

- 我的建议：
- 方案对比：
  1. ...
  2. ...
  3. ...
- 推荐方案：
- 主流程：
- 改动范围：
- 验证方式：

## 常见抽象的选择方式

选择承接逻辑的位置时，优先用最轻的抽象：

- **helper / shared module**  
  用于可复用的纯逻辑

- **service / manager**  
  用于流程编排、I/O、资源处理、多步骤执行

- **hook / controller**  
  用于带状态的交互式流程

- **adapter / bridge**  
  用于兼容边界、旧接口、第三方能力隔离

不要按习惯硬套，要按问题本身选。

## 不要这样做

避免给出以下类型的回答：

- 只有概念，没有实际改动指引
- 只列方案，不给推荐
- 为局部问题上来就大重构
- 为了“优雅”牺牲改动成本
- 虚构用户没要求的平台化蓝图
- 看起来很完整，但无法指导下一步实现

目标不是显得“很懂架构”，而是给出**下一步真的能落地的改法**。
