# React Hooks 精简规范版

> 目标：给团队内部快速统一认知。  
> 写法原则：只讲 **什么时候用 / 什么时候别用 / 最容易踩的坑 / 最小最佳实践**。  
> 不追求百科全书式覆盖，追求可执行。

---

## 1. useState

**用途**  
在函数组件里保存会驱动视图更新的本地状态。

**什么时候用**
- 组件内部的简单交互状态
- 表单值、开关状态、当前选中项
- 需要触发重渲染的数据

**什么时候别用**
- 能从 props / 其他 state 直接计算出来的值
- 跨很多层共享的状态
- 不需要驱动视图更新的值（这类更适合 `useRef`）

**关键规范**
- 只能在函数组件顶层或自定义 Hook 顶层调用
- 不要直接修改 state，必须通过 setter 更新
- 新状态依赖旧状态时，用函数式更新
- 初始值计算昂贵时，用函数形式传入初始值

**常见坑**
- 把可推导值也存成 state，导致冗余和不同步
- 直接改对象/数组内部字段，不触发正确更新
- React 18 下不要假设 `setState` 后能立刻拿到新值

**最佳实践**
- 状态按“是否一起变化”来拆分，不要机械地全拆碎
- 初始值类型尽量和后续更新保持一致

---

## 2. useEffect

**用途**  
处理副作用：请求、订阅、定时器、与外部系统同步。

**什么时候用**
- 发请求
- 注册/清理订阅
- 操作浏览器 API
- 和非 React 系统做同步

**什么时候别用**
- 纯计算
- 只是在 state/props 基础上派生值
- 能在事件处理函数里完成的逻辑

**关键规范**
- 依赖项必须包含 effect 中用到的组件作用域变量
- 清理函数不仅在卸载时执行，也会在下次 effect 执行前执行
- 不要把 effect 回调直接写成 `async`
- 不要在 effect 里无条件 `setState`，容易死循环

**常见坑**
- 漏依赖，导致读到旧值
- 一个 effect 里混多个不相关副作用，难维护
- 把本该在 render 阶段就能算出的值放进 effect 里二次同步

**最佳实践**
- 一个 effect 只处理一个副作用场景
- 高频触发请求时，再考虑防抖 / 节流 / 取消请求
- 如果依赖函数频繁变，可结合 `useCallback`，但别为了用而用

---

## 3. useRef

**用途**  
保存跨渲染周期持久存在、但**不会触发重渲染**的值；也常用于拿 DOM 引用。

**什么时候用**
- DOM 引用
- 定时器 ID
- AbortController
- 上一次的 props / state
- 不影响渲染的可变数据

**什么时候别用**
- 需要驱动界面更新的数据
- 本该交给 state 管的数据

**关键规范**
- `ref.current` 可直接改，但改了不会触发重渲染
- 初始值只在首次渲染生效
- DOM ref 在挂载前通常是 `null`
- 不要在 render 阶段随意改 `ref.current`

**常见坑**
- 用 ref 替代 state，结果数据变了 UI 不变
- 把 ref 当“隐藏全局变量”乱传

**最佳实践**
- ref 只存“持久但不驱动 UI”的东西
- 读写 ref 尽量放在事件回调或 effect 里

---

## 4. useImperativeHandle

**用途**  
配合 `forwardRef`，只向父组件暴露子组件必要的命令式能力。

**什么时候用**
- 聚焦输入框
- 清空输入
- 滚动到某位置
- 父组件必须命令式触发子组件某个动作，且没有更合适的声明式方案

**什么时候别用**
- 普通数据传递
- 核心业务状态共享
- 父组件其实可以通过 props 完成控制的场景

**关键规范**
- 必须配合 `forwardRef`
- 只暴露必要方法，遵守最小暴露原则
- 有依赖时要正确写依赖数组，避免闭包旧值
- 不要直接把整个 DOM 或内部实例裸暴露出去

**常见坑**
- 把子组件内部状态直接暴露给父组件
- 用命令式 API 代替本可声明式表达的逻辑

**最佳实践**
- 暴露的方法名要有明确语义：`focus`、`clear`、`scrollToTop`
- 命令式能力只解决“命令式交互”，不解决业务状态流

---

## 5. useLayoutEffect

**用途**  
在 DOM 更新后、浏览器绘制前同步执行副作用，常用于读布局、立刻改布局，避免闪烁。

**什么时候用**
- 读取布局信息后要立刻同步修正 UI
- 必须在绘制前完成 DOM 调整，避免视觉跳动

**什么时候别用**
- 普通请求
- 一般副作用
- 耗时计算

**关键规范**
- 它是同步执行的，会阻塞绘制
- 绝大多数场景优先用 `useEffect`
- SSR 场景下要谨慎，服务端不会执行它

**常见坑**
- 把重逻辑放进去导致卡顿
- 明明 `useEffect` 就够用，却滥用 `useLayoutEffect`

**最佳实践**
- 仅用于“无闪烁”要求很强的布局类副作用
- 逻辑越短越好

---

## 6. useInsertionEffect

**用途**  
为 CSS-in-JS 这类样式库在更早阶段插入样式规则准备的专用 Hook。

**什么时候用**
- 你在写 CSS-in-JS 库或底层样式能力

**什么时候别用**
- 普通业务组件
- 普通页面逻辑
- 一般 DOM 副作用

**关键规范**
- 执行时机早于 `useLayoutEffect`
- 不要在里面触发状态更新
- 需要清理插入的样式，避免污染

**常见坑**
- 业务代码误用
- 在里面做普通副作用逻辑

**最佳实践**
- 团队普通业务开发基本可以忽略这个 Hook
- 不是样式库开发，就别用

---

## 7. useDeferredValue

**用途**  
延迟非关键渲染，让关键交互先响应。

**什么时候用**
- 输入框联动长列表
- 大型过滤结果展示
- 非关键区域渲染明显卡顿

**什么时候别用**
- 没有明显卡顿
- 问题根源不在渲染，而在请求或重计算本身

**关键规范**
- 它延迟的是“渲染优先级”，不是原始 state 更新
- 主要优化的是复杂渲染，不是副作用开销
- 延迟值最终会追上原值，只是中间短暂滞后

**常见坑**
- 没有配合 `memo` / `useMemo`，优化效果很差
- 以为它能替代防抖或替代请求优化

**最佳实践**
- 只在“确实卡”的地方用
- 经常和 `memo`、`useMemo` 配合使用

---

## 8. useContext

**用途**  
跨组件层级传数据，避免层层 props 透传。

**什么时候用**
- 主题
- 当前用户信息
- 语言环境
- 多层共享但又不想一路传 props 的数据

**什么时候别用**
- 父子直传就能解决的问题
- 高频变化、粒度很细、性能敏感的大量状态

**关键规范**
- Provider 的 `value` 变了，消费组件会重新渲染
- 默认值只在没有对应 Provider 时生效
- 不要把所有共享数据都塞进一个超大 Context

**常见坑**
- 一个 Context 塞太多内容，导致大面积重渲染
- `value` 每次 render 都新建对象，导致无意义更新

**最佳实践**
- 按功能拆分多个小 Context
- `value` 需要时用 `useMemo` 稳定引用
- 数据复杂时，可结合 `useReducer`
- 如果团队已有 `use-context-selector`，优先用它做细粒度订阅

---

## 9. useCallback

**用途**  
缓存函数引用，避免函数每次 render 都变。

**什么时候用**
- 函数要作为 props 传给 `memo` 包裹的子组件
- 函数要作为其他 Hook 的依赖项
- 函数引用稳定本身就有价值

**什么时候别用**
- 普通函数只在当前组件内部使用
- 子组件没做 `memo`
- 没有性能问题却到处机械包 `useCallback`

**关键规范**
- 依赖项必须完整
- `useCallback` 本身也有开销，不是越多越好
- 它缓存的是函数引用，不是函数执行结果

**常见坑**
- 以为包了 `useCallback` 就一定更快
- 子组件没 `memo`，结果白包
- 漏依赖导致闭包旧值

**最佳实践**
- 只在“引用稳定真的有收益”时使用
- 先确认性能瓶颈，再优化

---

## 10. useMemo

**用途**  
缓存计算结果，避免重复执行昂贵计算；也常用来稳定对象/数组引用。

**什么时候用**
- 列表过滤、排序、大计算
- 给 `memo` 子组件传稳定对象/数组
- 给 Context Provider 提供稳定 `value`

**什么时候别用**
- 计算本身很轻
- 只是为了“显得专业”到处包一层

**关键规范**
- 依赖项必须完整
- `useMemo` 只做纯计算，不做副作用
- 它缓存的是结果，不是过程

**常见坑**
- 在 `useMemo` 里写副作用
- 轻量计算也强行缓存，得不偿失
- 漏依赖，拿到旧结果

**最佳实践**
- 只缓存真正昂贵或确实需要稳定引用的结果
- 计算函数保持纯净

---

## 11. useId

**用途**  
生成 SSR/CSR 一致的 DOM 唯一标识，避免 hydration ID 不匹配。

**什么时候用**
- `label` 和 `input` 关联
- `aria-describedby`
- 组件内部一组相关 DOM 元素的唯一标识

**什么时候别用**
- 列表 `key`
- 业务主键
- 接口参数

**关键规范**
- 它适合 DOM 标识，不适合业务语义
- 同一组件里多次调用会生成稳定但不同的 ID
- 生命周期内保持稳定

**常见坑**
- 拿它当列表 key
- 拿它当数据库/接口 ID

**最佳实践**
- 专门用于 DOM 可访问性和 SSR 一致性问题
- 列表 key 优先用真实业务 ID

---

## 12. useOptimistic

**用途**  
做乐观更新：先让 UI 看起来成功，再等待异步结果确认。

**什么时候用**
- 点赞
- 评论发送
- 列表添加/删除
- 成功率高、回滚成本低的异步交互

**什么时候别用**
- 支付、转账
- 高风险不可逆操作
- 失败后难以回滚的场景

**关键规范**
- 乐观状态不等于真实状态
- 请求失败时必须考虑回滚
- 更新函数应保持纯净

**常见坑**
- 只做“先展示”，没处理失败回滚
- 乐观结果和真实结果差异太大，导致明显闪烁

**最佳实践**
- 成功率高的交互优先考虑
- 必须设计失败恢复路径
- 可结合 loading/pending 提示减少误解

---

## 13. useSyncExternalStore

**用途**  
把 React 之外的外部数据源安全同步进组件。

**什么时候用**
- 订阅浏览器 API
- 对接外部 store
- 监听 `localStorage`、`MediaQueryList`、`ResizeObserver` 这类外部来源

**什么时候别用**
- 纯 React 内部状态
- `useState/useReducer/useContext` 就能解决的问题

**关键规范**
- 订阅函数要正确返回取消订阅函数
- 获取快照的函数应是纯函数
- SSR 时要考虑服务端快照

**常见坑**
- subscribe / unsubscribe 写不对，造成泄漏
- 快照函数不纯，行为不稳定

**最佳实践**
- 封装成自定义 Hook 对外使用
- 让组件只关心结果，不关心底层订阅细节

---

## 14. useTransition

**用途**  
把“非紧急更新”标记为 transition，让输入、点击等紧急交互优先响应。

**什么时候用**
- 切换筛选条件导致长列表重渲染
- 搜索输入后要更新复杂结果区
- 页面局部切换带来明显渲染压力

**什么时候别用**
- 普通小状态更新
- 没有卡顿的问题
- 试图拿它表示请求 loading

**关键规范**
- `isPending` 表示过渡更新是否在进行，不等于请求状态
- `startTransition` 里应放非紧急更新
- 更新逻辑要能承受中断和重试

**常见坑**
- 把输入框本身的值更新也放进 transition，导致输入变迟钝
- 用它替代真实的异步状态管理

**最佳实践**
- 紧急状态正常更新，重渲染重的部分放 transition
- 只在有明显交互卡顿时上

# 团队统一原则

1. **先分清“是否驱动 UI”**
   - 驱动 UI：优先 `useState`
   - 不驱动 UI：优先 `useRef`

2. **先分清“是不是副作用”**
   - 是副作用：`useEffect` / `useLayoutEffect`
   - 不是副作用：不要硬塞进 effect

3. **先分清“是不是性能问题真的存在”**
   - 没有实际卡顿，不要滥用 `useMemo` / `useCallback` / `useDeferredValue` / `useTransition`

4. **先分清“是不是共享状态真的跨层”**
   - 就近 props 能解决，不要先上 Context
   - Context 要拆小，不要搞巨型 Provider

5. **命令式能力永远是例外**
   - `useImperativeHandle` 只在确实需要时用
   - 业务主流程优先声明式表达

---

# 最后一句

**Hook 不是越多越高级，边界清楚、依赖正确、行为稳定，才是团队里真正好维护的写法。**
